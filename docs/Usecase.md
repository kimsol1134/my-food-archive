# 📱 [마이 맛집 아카이브] 유스케이스 명세서 (최종본)

## UC-01: 전체 맛집 갤러리 조회
* **주요 행위자:** 사용자
* **선행 조건:** 앱이 실행되어 홈 화면(`HomeScreen`)에 진입한 상태여야 한다.
* **단계별 시나리오 (기본 흐름):**
    1. 시스템은 기기의 로컬 데이터베이스에서 저장된 `ArchiveItem` 목록을 불러온다.
    2. 시스템은 불러온 데이터를 최신순으로 정렬하여, 화면 본문에 사진 썸네일을 바둑판(Grid) 형태로 나열한다.
* **예외 상황 (대안 흐름):**
    * **1-a. 데이터 없음 (Empty State):** 로컬 DB에 저장된 기록이 0개일 경우, 시스템은 화면 중앙에 "아직 저장된 맛집이 없어요. 하단의 + 버튼을 눌러 첫 기록을 남겨보세요!"라는 텍스트와 일러스트를 표시한다.

## UC-02: 키워드 기반 맛집 검색
* **주요 행위자:** 사용자
* **선행 조건:** 홈 화면에 1개 이상의 데이터가 렌더링되어 있어야 한다.
* **단계별 시나리오 (기본 흐름):**
    1. 사용자가 홈 화면 상단의 검색창(`TextField`)에 키워드를 입력한다.
    2. 시스템은 텍스트가 입력될 때마다 실시간으로 로컬 DB의 `searchKeyword` 필드를 탐색하여 일치하는(`contains`) 항목만 필터링한다.
    3. 시스템은 필터링된 결과값으로 갤러리 뷰를 즉시 업데이트한다.

## UC-03: 맛집 기록 추가 및 AI 분석 (✨사진 재선택 기능 포함)
* **주요 행위자:** 사용자
* **선행 조건:** 홈 화면에서 우측 하단의 FAB(+ 버튼)를 탭한 상태여야 한다.
* **단계별 시나리오 (기본 흐름):**
    1. 시스템은 사용자에게 OS 갤러리 접근 권한을 확인하고 갤러리를 호출한다.
    2. 사용자가 원본 음식 사진을 선택한다.
    3. 시스템은 선택된 사진을 들고 즉시 `AddEditRecordScreen` (생성 모드)으로 화면을 전환한다.
    4. 시스템은 화면 진입 즉시 로딩 스피너를 표시하고 화면 터치를 막는다.
    5. 시스템은 사진의 EXIF 데이터에서 촬영 날짜와 GPS 좌표를 추출하고, 역지오코딩을 통해 동/구 단위 지역명(예: 연남동) 텍스트로 변환한다.
    6. 시스템은 Gemini API를 호출하여 이미지로부터 메뉴명과 카테고리를 JSON 형태로 반환받는다.
    7. 분석이 완료되면 시스템은 로딩 스피너를 제거하고, 추출된 지역명, 메뉴명, 카테고리를 각 `TextField` 폼에 초기값으로 채워 넣는다.
    8. 사용자는 필수로 비어있는 '식당명'을 입력하고, AI가 분석한 데이터 중 틀린 부분이 있다면 텍스트를 수정한다.
    9. 사용자가 하단의 '저장' 버튼을 누른다.
    10. 시스템은 모든 입력 텍스트를 공백 없이 병합하여 `searchKeyword`를 생성한 후, DB에 저장(Insert)하고 홈 화면으로 복귀한다.
* **예외 상황 및 대안 흐름:**
    * **1-a. 권한 거부:** 갤러리 접근 권한이 없으면 "설정에서 사진 권한을 허용해 주세요"라는 스낵바를 노출하고 홈 화면에 머무른다.
    * **2-a. 초기 사진 선택 취소:** 갤러리를 닫으면 기록 추가가 취소되고 홈 화면에 머무른다.
    * **5-a. EXIF 데이터 부재:** 사진에 위치 및 날짜 데이터가 없는 경우 시스템은 크래시를 방지하고 해당 필드를 비워둔 상태로 폼을 제공한다.
    * **6-a. API 통신 실패:** 네트워크 오류나 타임아웃 발생 시, 시스템은 무한 로딩을 종료하고 "정보를 직접 입력해 주세요"라는 알림과 함께 비어있는 폼을 제공한다.
    * **8-a. ✨ 사진 재선택 (UX 개선):** 사용자가 입력 폼 상단의 '선택된 사진 썸네일 미리보기' 영역이나 '사진 변경' 버튼을 탭하면, 시스템은 다시 OS 갤러리를 호출한다. 사용자가 새로운 사진을 선택하면, **시나리오 4단계(로딩)부터 7단계까지를 다시 수행**하여 기존 폼의 데이터를 새로운 분석 결과로 덮어씌운다. 
    * **9-a. 식당명 미입력:** 식당명이 비어있으면 '저장' 버튼이 비활성화되어 저장을 막는다.

## UC-04: 맛집 상세 정보 보기
* **주요 행위자:** 사용자
* **선행 조건:** 홈 화면의 갤러리 뷰에 1개 이상의 사진이 표시된 상태여야 한다.
* **단계별 시나리오 (기본 흐름):**
    1. 사용자가 특정 맛집의 사진 썸네일을 탭한다.
    2. 시스템은 해당 기록의 고유 식별자(`id`)를 바탕으로 `DetailScreen`으로 이동한다.
    3. 시스템은 화면 상단에 원본 사진을 띄우고, 하단에 식당명, 메뉴명, 카테고리, 지역, 날짜를 텍스트 형태로 나열한다.

## UC-05: 기존 기록 수정
* **주요 행위자:** 사용자
* **선행 조건:** 상세 보기 화면(`DetailScreen`)에 진입해 있어야 한다.
* **단계별 시나리오 (기본 흐름):**
    1. 사용자가 상단 우측의 '수정' 아이콘을 탭한다.
    2. 시스템은 로딩 과정 없이 기존 데이터를 그대로 들고 `AddEditRecordScreen` (수정 모드)으로 이동하여 폼을 렌더링한다.
    3. 사용자가 입력 폼(`TextField`)의 텍스트를 원하는 대로 수정한다.
    4. 사용자가 '저장' 버튼을 누른다.
    5. 시스템은 수정된 텍스트로 `searchKeyword`를 재가공하고, DB를 업데이트(Update)한 후 홈 화면으로 이동한다.

## UC-06: 기록 삭제
* **주요 행위자:** 사용자
* **선행 조건:** 상세 보기 화면(`DetailScreen`)에 진입해 있어야 한다.
* **단계별 시나리오 (기본 흐름):**
    1. 사용자가 상단 우측의 '삭제' 아이콘을 탭한다.
    2. 시스템은 "정말 삭제하시겠습니까?"를 묻는 확인 다이얼로그 팝업을 띄운다.
    3. 사용자가 팝업에서 '삭제'를 승인한다.
    4. 시스템은 로컬 DB에서 해당 `ArchiveItem`을 완전히 삭제(Delete)한다.
    5. 시스템은 홈 화면으로 복귀하여 갱신된 리스트를 보여준다.
